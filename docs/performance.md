# 性能計測記録

計測環境: Apple Silicon (arm64) / macOS 26 / release ビルド / decodebench

## Phase 2: DecodeKit（2026-07-03）

要件: キャッシュミス時 M262 300ms以内 / Q3 100ms以内（Q3はJPEG抽出パスが主経路のためraw経路は参考値）

### 結果（read + LJ92 + ハーフサイズの合計）

| 機種 | read (pread) | LJ92デコード | ハーフサイズ(並列) | 合計 | 判定 |
|---|---:|---:|---:|---:|---|
| M262 (24MP) | 4〜8ms | 230〜250ms | 9〜19ms | **246〜276ms** | ✔ 300ms以内 |
| Q3 18MP (参考) | 3〜7ms | 160〜168ms | 7〜12ms | **176〜194ms** | （主経路はJPEG抽出） |

- 全21サンプルでデコードエラーなし。LJ92出力は libraw と全画素一致（テストで固定）
- LJ92 は約24Mサンプル/230ms ≒ 104Mサンプル/秒（単スレッド。リスタートマーカーが無いため並列化不可）

### 最適化の履歴（計測ベース）

| 変更 | M262 LJ92 | 備考 |
|---|---:|---|
| 初版（クロージャ内ネスト関数） | 298ms | |
| ensure32一括化+modulo除去（キャプチャあり） | 525ms | **悪化**。mutable varのクロージャキャプチャでボックス化 |
| キャプチャ排除（inout引数の静的関数へ） | 270ms | |
| Huffman LUT共有（256KB→128KB）+予測器1特化 | 230ms | 両機種とも全成分同一テーブル・予測器1 |

I/O:

- pread一括読み（rawDataContiguous）で 4〜8ms に短縮 → デコードと直列でも要件内

## Phase 3: 最小UI（2026-07-04）

アプリ実機計測（`LeicaSelect --folder samples --autotest`、Release、キャッシュなし逐次表示）。
レイテンシ = ナビゲーション開始 → CAMetalLayer present 完了。

| 機種 | 表示レイテンシ | 内訳: デコード | 内訳: テクスチャ+present |
|---|---:|---:|---:|
| M262 (11枚) | 267〜314ms | 238〜285ms | 約25〜40ms |
| Q3 (10枚) | 67〜200ms | 36〜161ms | 約30〜45ms |

- 全21枚 平均204ms / 最大313ms（先読みなしの素の値 = Phase 5 のベースライン）
- Q3 は JpgFromRaw の ImageIO デコード（JPEGサイズ 1.4〜3.8MB に比例して36〜161ms）
- M262 の decodebench との差分（デコード238ms vs 230ms）はRGBAコピーとアプリ環境分
- 判明した罠: SwiftUI の状態更新で古いフレームが再presentされ計測が誤発火
  → present通知にフレームidを付けて世代ガードで解決（コミット参照）

## Phase 5: CacheKit 先読み（2026-07-04）

autotest 往復2パス（往路=コールド+先読み、復路=全キャッシュヒット）。
glass = drawable.presentedTime 基準の実提示時刻 / app = キー→コマンドバッファcommit完了。

| 計測 | 値 | 備考 |
|---|---:|---|
| ヒット時 app 時間 | **0.1ms** | 16ms要件のアプリ側は余裕で達成 |
| ヒット時 glass 時間 | 32.8ms | 60Hz の vsync+コンポジタ2フレーム（OS律速） |
| 往路平均（先読み効果込み） | 115ms | 先読みなしの204msから改善。前方prefetch進行中に到着するため |
| キャッシュ常駐 | 861MB / 上限2GB | 全21枚が追い出しなしで常駐 |

- **16ms要件の解釈**: キー押下から描画コマンド提出までアプリ側0.1ms。
  以降の32.8msは macOS コンポジタのダブルバッファリング（present→glass に2vsync）で
  ウィンドウアプリでは原理的に消せない。120Hz(ProMotion)環境なら glass ≈16ms
- 試行錯誤の記録: actorホップ排除（ホットミラー）→効果なし、二重エンコード排除→効果なし、
  presentedTime での計測精密化 → app 0.1ms が判明し、残余がOS由来と確定
- 先読みポリシー: 前方2枚・後方1枚、移動のたびに窓外のプリフェッチをキャンセル
- **計測の注意**: 完全に隠れたウィンドウは macOS が present をドロップし
  presentedTime=0 になる。autotest の glass 計測はウィンドウ可視が前提

## Phase 8: プログレッシブ表示 + ダウンサンプルデコード + 先読み拡大（2026-07-06）

autotest（samples 21枚、v0.1.0比）。**PASS1の意味が「初回視認までの時間」に変わった**
（ミス時は埋め込みプレビュー速報が先に出て、本デコード完了時に差し替わる）。

| 計測 | v0.1.0 | Phase 8 | 変更 |
|---|---:|---:|---|
| PASS1 初回視認 avg | 90.6ms | 49.8〜55.7ms | ミス時速報（M262: 1472×976を8〜14msでデコード） |
| PASS1 max | 290.8ms | 116〜233ms | M262ミスの体感は約33ms（vsync律速）に |
| PASS2 全ヒット | 32.8ms | 32.7〜32.9ms | 維持（ホット窓±4で復路も全カバー） |
| キャッシュ常駐 | 861MB | 548MB | Q3テクスチャ 74MB→約28MB（画面長辺まで縮小） |

- 表示用デコードは最大スクリーン長辺を目標に ImageIO 縮小デコード（DCTスケーリング）。
  等倍(Z)時のみ full 経路で全画素。scene寸法は原寸のままなので座標系は不変
- 先読みは進行方向適応で前方4・後方1（M262のデコードは画像間並列が効く）
- **計測の注意（2026-07-06発覚）**: autotest はバイナリ直接 exec だとウィンドウが
  生成されず無限ハングする（/Applications に Developer ID 署名版を導入後に発生、
  ad-hoc署名のDerivedDataビルドで再現）。必ず
  `open -n -W --stdout <log> --args --folder <dir> --autotest` で起動すること

## Phase 7: フルデモザイク + 等倍 + グリッド（2026-07-04）

| 計測 | 値 | 備考 |
|---|---:|---|
| フルデモザイク (M262 24MP, bilinear, 行並列) | 62.7ms | 等倍要求時のみの遅延実行 |
| 等倍用フルパイプライン (read+LJ92+デモザイク) | 313ms | 以後はLRUキャッシュ(FrameKey full=true) |
| フル解像テクスチャ | 95.7MB/枚(M262) | 2GB上限で約20枚常駐可 |

- Q3 は JpgFromRaw テクスチャ(74MB)が実質原寸なので等倍にデモザイク不要
- 等倍のプログレッシブ: ハーフサイズを2倍拡大で即時表示→デモザイク完了で差し替え
  （ジオメトリをシーン寸法基準にしたので差し替え時に位置が飛ばない）
- グリッドは表示セルのみ遅延サムネ生成（埋め込みプレビュー→ImageIO、LRU 320MB）。
  起動→グリッド表示開始はファイル列挙のみで、1000枚2秒要件は構造的に満たす
  （実測は21枚のみ。大量フォルダでの実測は実運用時に確認）
