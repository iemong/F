## Q3 の構造（DNG 1.4 / little-endian）

IFD0 + SubIFD ×3 の4層構成。全10枚で共通。

| IFD | 内容 | 解像度 | 形式 | サイズ実測 |
|---|---|---|---|---|
| IFD0 | サムネイル | 720×480 | JPEG (YCbCr 8bit) | 70〜90 KB |
| SubIFD | **raw 本体** | 5312×3526 | LJ92 (CFA 14bit) 単一ストリップ | 19.5〜20.6 MB |
| SubIFD1 | 中間プレビュー | 1620×1080 | JPEG (YCbCr 8bit) | 240〜340 KB |
| SubIFD2 | **ほぼ原寸 JPEG** (JpgFromRaw) | 5288×3518 | JPEG (YCbCr 8bit) | 1.4〜3.8 MB |

- SubIFD2 は raw より縦横 24×8px 小さいだけの実質原寸。3層とも sips でデコード検証済み
- LJ92 の SOF3 ヘッダ実測: precision 14bit, 3526 lines × 2656 samples/line × 2 components
  （Bayer 横2画素を2成分1組にパック。展開後 5312×3526）
- CFAPattern2: `0 1 1 2` = RGGB。DNGVersion 1.4.0.0
- 注意: サンプルは 18MP モードのみ。60MP / 36MP モードは未検証

## M Typ 262 の構造（big-endian!）

TIFF標準の階層は IFD0 + SubIFD の2層のみ。使えるプレビューは **MakerNotes の中** にある。

| 場所 | 内容 | 解像度 | 形式 | サイズ実測 |
|---|---|---|---|---|
| IFD0 | 極小サムネ | **160×120** | JPEG (YCbCr 8bit) | 6〜11 KB |
| SubIFD | **raw 本体** | 5984×4000 | LJ92 (CFA, BitsPerSample=16) 単一ストリップ | 23.6〜28.1 MB |
| MakerNotes tag 0x0300 | **最大プレビュー** | **1472×976** | JPEG | 126〜414 KB |

- raw は無圧縮換算 47.9MB → 実測 ~26MB なので **M262 もLJ92圧縮**（「M262は無圧縮DNG」ではなかった）
- LJ92 SOF3 ヘッダ実測: precision **14bit**（コンテナは16bit宣言）, 4000 lines × 2992 samples/line × 2 components
- MakerNotes の位置: ExifIFD → tag 0x927C (MakerNoteLeica7) → 先頭8バイト `LEICA\0\x02\xff` → **big-endian の内部IFD** → tag 0x0300 が JPEG 実体
- exiftool は内部IFDエントリ2 (0x0302) で minor warning を出す（境界外参照）。自前パーサは 0x0300 だけ読めばよい

## 仮説の判定

### 仮説A: Q3 は原寸級の埋め込み JPEG を持つ → **成立**

SubIFD2 の JpgFromRaw が実質原寸。**Q3 は抽出＋JPEGデコードだけでセレクト用途が完結**。
100%等倍のピント確認もカメラ内現像品質のこのJPEGで実用上足りる見込み。

### 仮説B: M262 の埋め込みは数KBの極小サムネのみ → **概ね成立**

TIFF標準の場所には 160×120 (6〜11KB) しかない。ただし MakerNotes に 1472×976 が隠れており、
これは「グリッドサムネ」と「キー送り直後のファーストペイント」には使える。
一方、Retinaでのフィット表示（例: 5K = 5120×2880）にも 100%等倍にも解像度不足
→ **M262 には LJ92 デコード + ハーフサイズ縮約の高速パスが必要（当初想定どおり DecodeKit を作る）**。

## 実装への示唆

1. **DNGKit は両エンディアン対応が必須**（Q3=II / M262=MM）。TIFFヘッダのマジックで分岐
2. **LJ92 デコーダは1本で両機種をカバー**: どちらも SOF3 / 14bit / 2成分Bayerパック / 単一ストリップ。
   実測ヘッダ値（Q3: 2656×3526×2、M262: 2992×4000×2）をユニットテストのリファレンスにする
3. DNGKit の抽出APIは3種:
   - Q3: SubIFD2 (JpgFromRaw) / SubIFD1 / IFD0 — StripOffsets+ByteCounts の切り出しのみ
   - M262: MakerNotes 0x0300 の JPEG — ExifIFD→MakerNotes内部IFD の走査が必要
4. 表示パイプライン（機種別プログレッシブ）:
   - Q3: グリッド=IFD0 → 送り直後=SubIFD1 → 確定=SubIFD2原寸（デコード不要）
   - M262: グリッド=MakerNotes 1472×976 → 送り直後=同JPEG拡大表示 → 確定=LJ92ハーフサイズ縮約(2992×2000) → 等倍要求時=フルデモザイク
5. 性能要件の見通し:
   - Q3 キャッシュミス100ms: 2〜4MB read + HWデコードで余裕あり
   - M262 キャッシュミス300ms: 26MB read + LJ92デコード(24MP)。ここが最大のリスク。
     LJ92は逐次予測で並列化しにくい → ベンチマーク先行で実装（Phase 3で計測必須）
6. Orientation 適用を表示層の初期実装から入れる（縦位置サンプルで検証可能）
